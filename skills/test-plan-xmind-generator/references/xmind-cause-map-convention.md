# XMind 因果图约定

本 skill 生成的是测试导向的因果图，不是纯需求树。

## 层级结构

- 根节点使用项目名。
- 二级节点只放模块列表，例如“数据集选择”“映射编辑”“导出与修改记录”。
- 每个模块下面先放需求节点。
- 每个需求下面再向下衍生测试说明，推荐使用：
  - `测试点`
  - `注意项`
  - `异常`
  - `依赖/风险`

## 节点风格

- 节点文字保持简短、可测试。
- 组织分支时优先采用“需求 -> 测试说明”的阅读顺序，不要把测试点、异常、依赖拆成与需求平级的大分支。
- 二级节点尽量只保留模块，不要在二级节点直接放“测试点”“异常”“风险”等分类词。
- 某个需求下没有补充说明时，用兜底节点 `未提供`，不要直接省略。
- 节点默认使用中文通用测试概念，不直接使用某个项目内部字段名、表名或接口路径作为二级节点。
- 必须保留项目专有术语时，应只出现在必要的叶子节点中。

## 推荐内容

- 需求节点：写成清晰的需求条目，而不是模块摘要。
- `测试点`：校验逻辑、状态变化、角色行为、结果一致性。
- `注意项`：主流程、接口/数据口径、跨页面联动、实现约束。
- `异常`：空态、非法输入、服务失败、超时、边界值。
- `依赖/风险`：上游依赖、环境限制、版本冲突、外部接口稳定性。
